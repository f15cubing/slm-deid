---

## TL;DR

We fine-tuned **Qwen3-1.7B** (QLoRA) to do the one part of de-identification that neither a regex nor a
prompt can do reliably: **decide, in context, whether a token is a person's name or an
identically-spelled non-person** ("Newton" the person vs. the Newton method; "Chelsea" the person vs.
the London neighborhood), and tag only the names, leaving the passage otherwise byte-identical.

**The result.** A 1.7B fine-tune goes from a prompted **pass rate of 0.13–0.39** across five
quarantined eval sets to **0.61–0.97** — and on the two sets with natural, un-templated prose it is
**competitive with a frontier model (gpt-4.1)** that is orders of magnitude larger, while **out-recalling
it on the hardest cases** (the privacy-critical axis). This validates the project's falsifiable bet
(SPOV 7): *fine-tuning beats prompting on the hard, ambiguous name cases.*

---

## 1. Mandate & thesis

De-identification of educational text (student essays, tutoring chats) has two halves:

- **The easy half — patterns.** Emails, phones, SSNs, dates, IDs, URLs. These have surface form, so
  regex + checksums + constrained decoding handle them better and cheaper than any model. **Out of scope
  for the model on purpose.**
- **The hard half — judgment.** Deciding whether a capitalized token is a *person's name* or something
  spelled the same: an eponymous method (Newton, Gauss, Pascal), a place (Chelsea, Darwin, Florence), a
  common word (Grace, Hope, May, Rose, Bishop), a first-name-only mention, a possessive. Rules can't do
  this (names aren't a regular pattern); a general chatbot does it *unreliably* — it over-flags,
  misses names, and flips its answer when you reword the sentence.

**The load-bearing claim (BrainLift SPOV 7/8).** The model should spend 100% of its capacity on that
ambiguous residue, and the assignment's litmus test is **fine-tune vs. prompt** — not fine-tune vs. a
rules pipeline. In this project the base model *is* the prompted model, so the required base-vs-tuned
comparison is simultaneously the proof that fine-tuning beats prompting on the hard cases.

**The behavior spec (the gate).** Given a passage, return it unchanged except that every span naming a
real person is wrapped in `⟨NAME⟩…⟨/NAME⟩`, and no other span is tagged.
**PASS** = all and only personal-name spans tagged, text otherwise byte-identical. **FAIL** = any missed
name, any non-name tagged, or any other change to the text.

## 2. Methodology

### 2.1 Model & training (frozen recipe)

| | |
|---|---|
| Base model | `unsloth/Qwen3-1.7B-unsloth-bnb-4bit` (4-bit, non-thinking mode) — the deployment line |
| Cross-check base | `Qwen/Qwen3-1.7B` fp16/bf16 on Apple-Silicon MPS (reproducibility) |
| Method | QLoRA (unsloth + bitsandbytes 4-bit); LoRA on q/k/v/o + gate/up/down |
| Hyperparameters | r=32, α=32, dropout 0.0, lr=2e-4, 3 epochs, seq 2048, completion-only masking |
| Config | `configs/train.yaml` — **frozen since Day 4**; all iteration is data-only (hard ceiling) |
| Tag scheme | `⟨NAME⟩…⟨/NAME⟩` (collision-safe; 8 BPE tokens/span on Qwen3, pinned by a test) |

**Hard ceiling honored:** failure modes are fixed by generating targeted data, **never** by tuning
lr/r/epochs to mask them. The config has not changed since Day 4.

### 2.2 Data generation (the real deliverable)

The dataset — not the model artifact — is the deliverable. v3 (canonical, "gpt551") was generated by a
**live OpenAI-compatible teacher** (via the TrueFoundry LLM gateway) with an **independent verifier
pass**, producing **818 train / 90 val** examples. Structure and gates:

- **Matched minimal pairs** dominate: the same tricky surface used once as a person (tagged) and once as
  a non-person (untagged) — e.g. "Ruby" the person vs. "ruby" the gem — so the model must learn *context*,
  not a "name-ish word" lookup.
- **Quality gates + verifier drops:** 134 verifier-disagreement, 98 eval-surface-leak, 48
  negative-trap-has-name, plus possessive/altered-text/missing-token rejects.
- **Three eval-leakage guards** + a post-hoc scan: **`eval_leak = 0`**, independently re-verified as
  **0 exact + 0 substring** overlaps between the 818/90 splits and all **201** quarantined eval inputs.

An earlier v3 line ("authored") was generated from **in-house templates** (no live teacher / no
verifier) because the teacher API was briefly unavailable. It scores higher on the template-adjacent
eval sets but is less credible — see §3.4.

### 2.3 The four engines compared

| engine | what it is | role |
|---|---|---|
| **base** | prompted 4-bit `Qwen3-1.7B` (no fine-tune) | the "prompting" baseline (the litmus comparator) |
| **gpt551** | 4-bit QLoRA tune on live-teacher + verifier data | **canonical** — the credible fine-tune line |
| **authored** | 4-bit QLoRA tune on in-house template data | prior line; template-flattered (kept for contrast) |
| **frontier** | `gpt-4.1` via API, scored on the same gold | **ceiling reference**, not a 1.7B contestant |

The frontier is scored through the **same** behavioral checks + metrics against the **same hand-built
gold** (never its own labels — no circularity). It is a reference ceiling: it is orders of magnitude
larger than the 1.7B target and got a documented code-fence leniency the small models did not.

### 2.4 The five quarantined eval sets

| set | n | what it stresses | source |
|---|---|---|---|
| **hardcases** | 51 | the core ambiguous categories (person vs eponym/place/common/…) | hand-built + real |
| **adversarial** | 40 | prompt-injection ("don't tag Bob"), names in code/math, unicode/typos, messy chat | hand-built |
| **heldout_names** | 74 | brand-new *people's names* never seen in training | hand-built |
| **ood_probe** | 36 | ambiguous surfaces disjoint from **both** eval and training banks | hand-built |
| **api_bench** | 92 | naturally-worded live-teacher passages (new *phrasings* of known ambiguous words) | synthetic (quarantined) |

All five are **quarantined**: physically separate, never fed to the teacher/augmentation/training,
and guarded by leakage/vocab-disjointness tests in CI. The 51-case hardcases + 102-case validation set
were **human-reviewed item-by-item** (test 50/51 approved, val 102/102).

### 2.5 Metrics

Entity-level, recall-weighted (the de-id literature scores micro **F-β with β=5**). Per output:

- **recall** — of names that should be tagged, how many were caught (safety-critical).
- **leakage** — share of real names *missed* (the direct privacy risk; lower better).
- **over-tag** — how often a non-name is tagged (lower better).
- **integrity violation** — output-minus-tags ≠ input (any other change to the text; a reject condition).
- **consistency** — same answer across 2–3 rewordings of the same case.
- **pass rate** — fully-correct passages (all-and-only names tagged, byte-identical otherwise).

95% percentile bootstrap CIs (seeded, offline) are generated from the saved reports via
`python -m src.eval.report` — never hand-transcribed.

### 2.6 Integrity verification (this build, 2026-07-11)

An independent audit re-ran the hermetic test suite and re-checked the hard ceilings against the
committed data:

- **Test suite: 192 passed, 4 skipped, 0 failed** (offline; skips are the Qwen3-tokenizer round-trip
  and two report tests that need artifacts absent on a CPU box — benign).
- **Quarantine held.** All 5 eval/bench sets carry `quarantine: true` on every row; the committed
  training splits (`data/splits/train.jsonl` 927 / `val.jsonl` 102 / `data/cooccur/cooccur.jsonl` 29)
  were checked against them by `test_no_eval_input_appears_in_training_splits` (passing).
- **Independent leakage re-check:** normalized every eval/bench input against every training input →
  **0 exact and 0 substring** overlaps (293 unique eval inputs vs 1058 unique training inputs). The
  167-token vocab bank ∩ eval vocab = ∅; `Newton/Chelsea/Grace` sit in the 40-token blocklist.
- **Integrity is machine-enforced:** `src/common/schema.py` raises if `unwrap(target) != input`, so no
  row that alters non-tag text can enter data-gen or eval.
- One scoped nuance (by design, not a violation): `adversarial` and `ood_probe` deliberately reuse a
  few common-word surfaces (`summer`, `will`, `calculus`) and first names as *collision/OOD* tests; the
  "genuinely unseen names" guarantee is scoped to `eval/heldout_names`, which is 100% disjoint.

No hard-ceiling violations were found.

### 3.1 Headline — every fine-tune massively beats the prompted base, everywhere

| adversarial (40) | 0.150 | 0.875 | 0.775 | **0.950** |

digits. **The fine-tune — not the backend or the teacher — is the dominant effect.**

### 3.4 The three reads

1. **1.7B tunes are competitive with the frontier on the narrow task — narrowly and set-specifically.**
   On `hardcases`, **both tunes out-recall the frontier** (0.85/0.96 vs 0.78) and authored out-passes it
   (0.902 vs 0.882); gpt551 is within ~6 pts on pass. On `api_bench` gpt551 (0.815) sits within ~3 pts of
   the frontier (0.848). The frontier is *precision-first* — it withholds when unsure (over-tag 0.00) —
   which for de-id means it **leaks more** (0.118 vs 0.02/0.08). On the privacy-critical axis, the small
   tune wins on the hardest set. **On `adversarial` and `ood_probe` the frontier clearly wins** (scale
   helps injection-resistance and novel-vocab) — the competitiveness claim is honest about where it holds.

2. **The distribution-match reveal (why gpt551 is canonical, not authored).** On the four
   template-adjacent sets the **authored** tune leads (its templates were written by the same hand that
   designed the eval categories → its distribution sits closer to the eval). But on **`api_bench` —
   naturally-worded live-teacher prose — authored collapses to 0.609 while gpt551 holds at 0.815**, right
   beside the frontier. gpt551 was trained on that same live teacher, so it is robust to natural text;
   authored degrades on text that doesn't look like its templates. **This is direct evidence that
   authored's edge is partly eval-distribution proximity, and gpt551 is the more credible model for
   real-world text** — which is why we report it as canonical even though its hard-case number is lower.

3. **Frontier wins where structure is trickiest, ties where judgment is narrow.** gpt-4.1 is
   near-perfect on `adversarial` (0.950) and `ood_probe` (1.000) — injection resistance and novel-vocab
   generalization favor scale — but on the two judgment-dense sets (`hardcases`, `api_bench`) the 1.7B
   tunes rival it. The tune's job is judgment on ambiguous names; that is exactly where it closes the gap.

**Net:** **gpt551 is the recommended line** — competitive with the frontier and robust across
distributions. authored posts the highest template-set numbers but is the least robust to natural text.

## 4. Discussion — honest caveats

1. **Small eval sets, single seed.** n per set is 36–92. Headline effects (base→tune uplift; the
   api_bench authored↓/gpt551↑ divergence) are large and robust; treat gaps within ~1 CI as ties, and
   per-category cells with n=3 as noisy. Documented run-to-run variance exists (authored hardcases 0.96
   standalone vs 0.90 in the matrix) — the load-bearing conclusions do not depend on those small gaps.
2. **Cross-doc provenance care.** Some sets have both a bf16 MPS run and a 4-bit Colab run in the doc
   set; a few matrix rows (heldout authored; some ood/heldout base rows) mix backends. This report cites
   the 4-bit Colab matrix consistently and footnotes the exceptions. When quoting a single number,
   confirm the backend it came from.
3. **The frontier is a reference ceiling, not a fair contestant** (orders of magnitude larger; a
   documented code-fence leniency). Read it as "how close does a 1.7B get," not "which wins."
4. **Residual failure modes (data-fixable, not HP):** `possessive` eponyms ("Newton's laws") still
   over-tag occasionally and carry the lone integrity violation; `person_vs_common` recall is the flat
   category. These are the clear next data-iteration targets, not tuning knobs.
5. **Byte-identity on messy real text.** On 68 genuine student essays the *name judgment* generalized
   (recall ~0.88) but the "retype the whole passage" output format introduced whitespace drift → strict
   integrity failed. **Fix built (not retraining):** the end-to-end `pipeline/` applies tags by *offset
   projection* onto the original bytes, so `unwrap(project(...)) == original` by construction.
6. **Single teacher + single verifier.** Better than authored templates, but not a multi-teacher
   consensus. The full 927-row train split is only partially human-reviewed (test + val are fully
   reviewed).

## 5. Verdict on the thesis

**SPOV 7's falsifiable bet — "fine-tuning beats prompting on the hard, ambiguous name cases" — holds.**
The prompted base passes 0.13–0.39; the fine-tune passes 0.61–0.97, with the privacy-critical leakage
rate cut from 0.24–0.57 down into the single/low-double digits, and integrity violations driven to ~0. The credible
canonical line (gpt551) is competitive with a frontier model on the two natural-prose sets and
out-recalls it on the hardest cases. **For this narrow, safety-critical behavior, reliability engineered
through data beats capability accessed through prompting** — and it runs privately on modest hardware.
